Kamegor Project Plan

A refined vision for a location-based, P2P live video marketplace connecting Buyers with Sellers broadcasting from unique locations.

App Description (Refined)

Kamegor is evolving into a dynamic cross-platform mobile marketplace for live, peer-to-peer (P2P) video streams.

The core refinement centers around Seller activation and visibility. Users can explicitly opt-in via their profile to become "Sellers". Once activated and online, their profile picture appears as an interactive marker on the main map at their current location.

"Buyers" browse the map and can tap these markers to view a detailed Seller profile. This profile acts as a central hub, showcasing the Seller's description, rating, reputation tier, transaction history (initially DB-logged, exploring blockchain concepts for future trust enhancement), and crucially, their real-time status (Online / Streaming / Offline).

If a Seller is broadcasting live, Buyers can initiate a paid, per-minute viewing session directly from the profile, connecting via secure P2P WebRTC. The entire real-time ecosystem ‚Äì map updates, user presence, signaling, and transactions ‚Äì is powered by a robust Elixir/Phoenix backend.

Risks & Mitigation

‚ö†Ô∏è Privacy (Seller Location Exposure)

Showing persistent locations of all online Sellers is a significant risk.

Mitigation:

  • Require App in Foreground: Only show Sellers on map when app is actively used (MVP).
  • Location Obfuscation (Recommended): Show generalized location (e.g., neighborhood level) when 'Online', precise only when 'Streaming'.
  • Explicit Consent: Clear opt-in explaining visibility.
  • Future: Visibility controls for Sellers.

‚ö†Ô∏è Map Performance/Clutter

A high density of online Sellers could impact map usability and performance.

Mitigation:

  • Efficient Geospatial Queries (Backend Indexing).
  • Client-side Map Marker Clustering.
  • Limit data transfer (only viewport updates).

‚ö†Ô∏è State Management Complexity

Reliably tracking `offline`/`online`/`streaming` across network issues and app lifecycle events is challenging.

Mitigation:

  • Leverage Phoenix Presence robustly.
  • Careful client-side state logic.
  • Design for graceful reconnections and state synchronization.

‚ö†Ô∏è WebRTC Complexity & Reliability

Establishing reliable P2P connections across diverse networks and devices is inherently complex.

Mitigation:

  • Thorough STUN/TURN server configuration and testing.
  • Robust signaling logic and error handling.
  • Extensive testing on various devices and network types (NAT traversal).
  • Consider managed WebRTC services if self-hosting TURN becomes too burdensome.
